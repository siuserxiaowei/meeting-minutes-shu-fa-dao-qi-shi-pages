第一次对账与产品介绍页校准

产品商业化内容AI组织自动化

这场 39 分钟短会真正值得学的，不是费用表，而是一个项目从内部协作走向外部销售时的第一轮校准。逐字稿里连续出现域名、服务器、飞书链接、官网、公众号、产品介绍图、钉钉开放平台、OpenClaw/Hermes 表达边界，说明团队正在把“我知道怎么用”改写成“客户一眼能懂、能转发、能持续获得服务”。最大的增量是：对外材料不能把操作流程、产品定位、平台能力和售后服务混在一起；未跑通的能力必须写成测试中或可定制。

学习价值判断

值得学。它短，但是真实暴露了产品从内部 demo 走向客户材料时最容易犯的错：表达过满、边界不清、承接链断。不要学具体项目，学这套发布前校准方法。

维度短期长期判断依据
钱 3 4 短期只是改材料，长期会影响客户是否愿意转发、试用和继续问下去。
关系 4 4 飞书、官网、公众号的分工，本质是在降低合作方帮你推荐时的解释成本。
技能 5 5 产品页、能力矩阵、边界表达是所有早期项目都能复用的基础功。
影响力 3 4 如果每个项目都能沉淀成公开资料，个人影响力会从聊天变成可搜索资产。

公众号文章待发 售后服务正在从人肉答疑转向内容化承接。

核心故事

不买高价 AI 域名

冲突：看起来更像 AI 的域名反而更贵、更容易被对手盯上。

拆解：域名不是越显眼越好，早期项目要同时考虑成本、记忆点和被复制风险。

  • 选择更便宜的品牌域名
  • 把网站作为外部入口
  • 避免过早暴露可抄路径

飞书能打开，但客户要看官网

冲突：团队内部觉得飞书方便，合作方担心客户没有飞书或不适应。

拆解：内部协作工具只能降低团队成本，不能替代客户视角的销售页面。

  • 反复确认是否需要登录
  • 提出网页、单图、公众号三种承接
  • 客户转发场景被单独讨论

子 agent 不是第三个并列卖点

冲突：介绍图把 OpenClaw、Hermes、子 agent 并列，可能让客户误解产品结构。

拆解：技术表达要按职责层级讲清楚，否则越高级的词越容易降低信任。

  • OpenClaw 偏执行
  • Hermes 偏分析
  • 子 agent 应表达为团队协作能力

拆解口径

层级这次采用的判断标准参考
参考 SOP 按“拆解不是缩写”的标准，从逐字稿找故事、数字、边界和动作，而不是复述智能摘要。 本地参考文件夹 METHODOLOGY
公开边界 公开版只保留产品发布方法，不保留私密链接、长原话和具体内部凭据。 本站公开边界

主题地图

外部承接四件套

官网、单图、飞书资料页、公众号分别承担不同任务。

  • 官网讲对象与结果
  • 单图方便转发
  • 飞书保留协作资料
  • 公众号承接持续服务

平台能力矩阵

平台接入要按事实分层，不能把想做写成已做。

  • 已跑通
  • 测试中
  • 可定制
  • 暂不承诺

智能体表达

产品角色要从客户理解出发，而不是堆内部术语。

  • 执行
  • 分析
  • 任务拆分
  • 团队协作

发布前校准

每次对外发布前要过一次边界检查。

  • 客户是否能看懂
  • 是否能转发
  • 是否过度承诺
  • 下一步入口是否明确

道 底层原则、价值判断、长期认知。

核心判断
早期产品走向外部时，最重要的不是把功能说满，而是把客户能否理解、能否转发、能否持续获得服务说清。
拆解要点
  • 外部材料不是内部流程
  • 能打开不等于能成交
  • 边界表达会影响信任
  • 短会也能产生发布 SOP
关键证据
  • 飞书链接可看但仍讨论网页
  • 产品介绍页从流程转向产品

  2. 更新能力矩阵
  3. 公众号加服务栏目

可复用动作
发布前先做“客户可理解性”检查。

势 趋势、外部环境、机会窗口、结构性变化。

核心判断
趋势是企业协作入口正在被 AI 盒子和智能体重塑，但早期胜负仍在表达、信任和服务闭环。
拆解要点
  • IM 是新入口
  • 内容是售后系统
  • 官网是信任资产
  • 未验证平台不能抢跑
关键证据
  • 多个 IM 被纳入图示
  • 公众号被放进服务链
  • 官网第二天部署
行动清单
  1. 先做飞书闭环
  2. 再做钉钉 POC
  3. 同步沉淀公开资料
证据依据
会议把飞书、钉钉、Slack、公众号、官网都放在同一个交付链里讨论。
可复用动作
不要只盯技术接入，要先抢客户解释权。

个人学习抓手

方向我要练的能力练习方式
产品表达 把内部流程翻译成客户结果 每个产品页先写客户视角的一句话，再补功能。
边界管理 把已跑通和想做分开 能力矩阵按状态更新，不写模糊承诺。
发布 SOP 对外发布前做一次四入口检查 官网、单图、飞书、公众号各自验收。
技术命名 技术角色用职责解释 所有智能体名词后面都补一句人话。

章节索引

  1. T01 对账背后是资产登记费用表里讨论域名、服务器和后续支出，实质是把项目资产、成本和责任人从口头状态转成可追踪状态。
  2. T02 飞书链接能看，不等于适合销售讨论反复确认非飞书用户是否能打开，同时也意识到客户推荐场景更适合官网、单图和公众号组合。
  3. T03 操作流程不是产品介绍第一版资料偏流程，第二版被要求更直观、更适合给企业和中小客户转发。
  4. T04 公众号不是宣传口，是持续服务口文章、技巧、新 skill 和使用更新要沉淀在公众号，让购买后的服务可持续。
  5. T05 智能体关系必须说准确OpenClaw、Hermes 和子 agent 不能并列成误导结构，要表达为职责分层和团队协作。
  6. T06 钉钉接入只能写测试中钉钉需要企业和开放平台配置，还没跑通前不能写成默认支持。

行动项

  1. 把费用表拆成已支出、待补齐、待确认三类，并标注下一步负责人。
  2. 官网首屏只回答四件事：给谁用、解决什么、看到什么结果、怎么联系。
  3. 单图介绍页改成客户能转发的版本，减少内部术语和装饰性表达。
  4. 平台能力写成三列：已跑通、测试中、可定制，避免过度承诺。
  5. 把 OpenClaw、Hermes、子 agent 的关系改成职责层级，而不是并列口号。
  6. 公众号固定承接教程、FAQ、新 skill 和案例更新。
