业务发展及招人策略讨论

AI组织产品增长商业化Agent

这篇不是单纯聊招人，而是在讨论一个 AI 时代的小组织怎么运转：GitHub、TPR、阿里云、MCP、Codex、飞书话题群如何变成内部 Agent 工作流；产品和运营如何自助查数据；为什么要用低配资源训练人和 Prompt；老业务如何 P0 化维持现金流；新业务如何先由核心人突破；招人为什么看“道”而不是只看“术”。我自己学习这篇，最该沉淀的是组织级 Agent 经验：怎么把工具、上下文、角色和业务优先级组织成可执行系统。

主题地图

协作入口

飞书不是简单沟通工具，而是项目上下文、任务、信息源和 Agent session 的组织入口。

  • 一个话题一个群
  • 每天固定同步
  • 把判断放进同一空间
  • 减少跨工具丢上下文

Agent 内部工具

MCP、Codex、GitHub、云服务和内部数据可以组合成产品运营也能使用的工具链。

  • 数据自助查询
  • 排障辅助
  • 文档检索
  • 把技术能力产品化给内部用

低配训练法

低配资源不是失败条件，而是训练人和流程的压力测试。

  • Prompt 必须更清楚
  • 验收标准更重要
  • 流程不能靠强模型遮掩
  • 团队能力可迁移

第二曲线

老业务只维持 P0，核心资源投入新业务突破，避免现金流不稳定时还被旧流程拖住。

  • 老业务降噪
  • 上游解决复杂问题
  • 外包让利
  • 核心人先破局

招聘标准

招人看“道”：底层判断、业务意识、稳定性和成长性，比只会某个技术动作更关键。

  • 2-3 年经验的潜力层
  • 业务岗决策影响大
  • 6 个月真实观察
  • 用任务看判断力

推广观察

关键词流行度和展示占比用来判断新业务方向，不靠单次灵感拍脑袋。

  • humanizeAI
  • bypass
  • 搜索量
  • 曝光点击
  • 内容缺口

道法术器势拆解

道 底层原则、价值判断、长期认知。

核心判断
AI 时代组织的底层原则是：让人、Agent、上下文和业务优先级形成一个系统。真正稀缺的不是单个工具，而是团队能否持续把判断、信息和执行沉淀进同一条工作流。
拆解要点
  • 飞书第一入口的本质是统一上下文。
  • 低配资源训练的是人的表达、判断和验收能力。
  • 老业务只能占用 P0 资源，新业务必须由核心人先突破。
  • 招人看“道”，因为业务判断错误会造成大量资源沉没。
  • AI 边界必须明确，否则产品会被更强平台能力碾压。
关键证据
  • 材料中反复出现飞书话题群、一个话题一个 session 和上下文同步。
  • 低配模型被用来提升 Prompt 要求，而不是简单追逐强模型。
  • 老业务 P0 化和第二曲线被放在同一组经营判断里。
  • 招聘讨论明确强调看“道”而不是只看“术”。
行动清单
  1. 为团队写一条统一工作流。
  2. 每天同步上下文而不是只同步任务。
  3. 把老业务和新业务资源边界写清楚。
  4. 面试时用判断题而不是只用技能题。
证据依据
材料中反复出现飞书话题群、一个话题一个 session 和上下文同步。；低配模型被用来提升 Prompt 要求，而不是简单追逐强模型。；老业务 P0 化和第二曲线被放在同一组经营判断里。；招聘讨论明确强调看“道”而不是只看“术”。
可复用动作
我复盘任何 Agent 项目时，都要追问：这个 Agent 将改变团队哪条工作流，而不只是能回答什么问题。

法 可复用方法、流程、判断框架。

核心判断
这篇可沉淀成“Agent 组织法”：入口统一、上下文同步、工具接入、低配训练、P0 降噪、新业务突破、人才筛选。
拆解要点
  • 入口统一：飞书承接话题和长期上下文。
  • 上下文同步：每天固定输入信息源和判断。
  • 工具接入：MCP、Codex、GitHub、云服务和数据查询。
  • 低配训练：用限制逼出更好的 Prompt 和验收。
  • 经营取舍：老业务 P0，新业务核心突破。
  • 人才筛选：看底层判断和真实任务表现。
关键证据
  • GitHub、TPR、阿里云、MCP、Codex 被放在内部工具链里讨论。
  • 产品和运营自助查数据说明工具链不是只给工程师用。
  • 上游稳定、外包让利、老带新说明经营流程在重组。
  • 6 个月试用期说明人才判断需要真实任务周期。
行动清单
  1. 为每个项目建立一个话题群。
  2. 定义每日上下文同步格式。
  3. 列出 Agent 可调用工具和不可做事项。
  4. 每周复盘低配资源下的问题。
  5. 为老业务设置只处理 P0 的规则。
证据依据
GitHub、TPR、阿里云、MCP、Codex 被放在内部工具链里讨论。；产品和运营自助查数据说明工具链不是只给工程师用。；上游稳定、外包让利、老带新说明经营流程在重组。；6 个月试用期说明人才判断需要真实任务周期。
可复用动作

术 可执行动作、技巧、话术、步骤。

核心判断
可执行动作非常具体：建话题群、固定上下文输入、内部数据查询 Agent、低配模型训练、P0 清单、关键词观察表、面试判断题和 6 个月真实任务。
拆解要点

  • 每天把订阅信息源、思考和新发现同步进去。
  • 用 MCP/Codex/GitHub/云服务搭内部数据和排障工具。

  • 把 humanizeAI、bypass 等关键词纳入搜索观察。
  • 面试时让候选人解释判断过程。

  • 内部工具链包含 GitHub、TPR、阿里云、MCP、Codex。
  • 关键词推广围绕搜索量、展示占比、曝光点击展开。
  • 试用期和 2-3 年经验人才被具体讨论。

  2. 写一份每日上下文模板。
  3. 做一个产品运营可用的数据查询命令。
  4. 建立关键词观察表。
  5. 把面试问题改成“为什么这么判断”。
证据依据
材料中明确提到飞书作为第一入口。；内部工具链包含 GitHub、TPR、阿里云、MCP、Codex。；关键词推广围绕搜索量、展示占比、曝光点击展开。；试用期和 2-3 年经验人才被具体讨论。

器 工具、资源、模板、渠道、技术栈。

核心判断
器的重点是把工具接成组织基础设施：飞书承接上下文，GitHub 承接代码，MCP 承接工具调用，Codex 承接工程执行，云服务承接部署，搜索数据承接增长判断。
拆解要点

  • GitHub：项目代码、任务和版本沉淀。
  • MCP：把工具能力接入 Agent。
  • Codex：工程执行与代码协作。
  • 阿里云和内部平台：部署、数据和业务系统。
  • 关键词工具：观察搜索量、展示、点击和趋势。
关键证据

  • 飞书话题群被用作业务协作入口。
  • 关键词流行度和 ASA 展示占比被用来判断推广。
  • 产品和运营自助查数据说明工具要面向非工程角色。

  3. 把权限和业务边界写清楚。

可复用动作

势 趋势、外部环境、机会窗口、结构性变化。

核心判断
势在于组织形态正在变：强模型、Agent 工具、MCP 和低代码式协作让小团队能做更多事，但也会淘汰只靠重复执行的人和产品。机会属于能把业务判断、上下文和工具链组织起来的团队。
拆解要点
  • AI 会让产品运营也能执行部分数据和排障工作。
  • MCP 和 Agent 工具会把内部系统变成可调用能力。
  • 只做薄界面的产品更容易被平台能力覆盖。
  • 老业务现金流不稳定时，第二曲线更关键。
  • 人才价值从会某个动作转向会判断和组织资源。
  • 关键词数据会更早参与业务方向判断。
关键证据
  • 讨论里把 AI 协作、AI 组织边界和下个产品风险联系起来。
  • 老业务现金流与新业务突破被同时讨论。
  • 低配资源训练法说明强模型不是唯一竞争力。
  • 关键词流行度观察显示增长判断更数据化。
行动清单
  1. 把产品壁垒写成“模型不能直接替代的业务闭环”。
  2. 持续训练团队的判断和上下文整理能力。
  3. 用关键词和用户反馈捕捉新机会。
  4. 把老业务资源占用压到可控范围。
证据依据
讨论里把 AI 协作、AI 组织边界和下个产品风险联系起来。；老业务现金流与新业务突破被同时讨论。；低配资源训练法说明强模型不是唯一竞争力。；关键词流行度观察显示增长判断更数据化。
可复用动作
我的趋势判断：未来不是每个公司都有一个聊天机器人，而是每条业务流程里都有可调用的 Agent 节点。

个人学习抓手

方向我要练的能力练习方式
Agent 组织设计 练习把 Agent 从 demo 看成组织工作流的一部分。 每个 Agent 都写清入口、上下文、工具、执行、验收和复盘。
上下文管理 练习让团队和 Agent 共享同一套世界背景。 每天固定同步信息源、新发现、判断和任务变化到话题群。
低配训练 练习在低配模型或资源约束下提高 Prompt、流程和验收质量。 故意用低配资源跑一轮，记录哪些表达、步骤和验收标准不够清楚。
经营取舍 练习区分老业务 P0 维持、新业务突破和资源迁移条件。 写一张 P0 清单，只保留挂了、大故障、严重影响客户的事项。
招聘判断 练习从“道”看人：判断力、稳定性、业务意识和成长性。 面试时让候选人解释为什么这样判断，并用真实任务观察 6 个月。
增长观察 练习用关键词流行度、展示、点击和内容缺口判断方向。 围绕 humanizeAI、bypass 等关键词建立持续观察表。

章节索引

  1. T01 GitHub、TPR、阿里云、MCP、Codex 的内部工具链讨论把项目托管、云服务、Agent 协议和编码工具连接起来，让产品、运营也能自助查数据和处理问题。
  2. T02 飞书作为第一入口和话题群组织法用飞书承接信息、任务和上下文；一个话题一个 session，避免信息散在多个地方。
  3. T03 低配资源反而训练 Prompt 和流程能力强模型会掩盖人的能力，低配模型或资源不足时，反而会逼迫团队把 Prompt、流程和验收标准练出来。
  4. T04 每天同步信息源和思考，形成共享世界背景固定把订阅信息源、新发现、判断和上下文放进话题群，让 Agent 和人共享同一套背景。
  5. T05 老业务现金流和第二曲线老业务现金流不稳定，必须用新业务寻找第二曲线；老业务只保留 P0 级处理。
  6. T06 上游稳定、外包让利和老带新复杂问题尽量让上游解决，早期业务部分让利给外部伙伴或新人，用老带新降低核心团队负担。
  7. T07 新业务先由核心人突破新业务不应一开始就交给旧团队惯性执行，要先由核心人找到突破，再迁移合适成员。
  8. T08 AI 协作和 AI 组织边界讨论强调要确认 AI 的边界，否则下一个产品可能被更强模型或平台能力直接碾压。
  9. T09 招人看“道”，不是只看技术动作2-3 年经验的人如果已经出现底层判断和业务意识，可能比只会执行的人更值得培养。
  10. T10 业务岗决策成本和 6 个月试用业务岗错误决策会沉没很多资源，因此要用更长观察周期和真实任务判断。
  11. T11 关键词流行度、展示占比和推广观察围绕 humanizeAI、bypass 等关键词，讨论搜索量、曝光点击和展示占比，用数据判断推广方向。

行动项

  1. 把飞书设为项目第一入口：一个业务话题一个群，一个群对应一个长期 session。
  2. 每天固定同步订阅信息源、业务思考、新发现和判断，让团队与 Agent 共享上下文。
  3. 优先把数据查询、排障、文档检索做成内部 Agent 工具，让产品和运营自助使用。
  4. 故意保留一部分低配模型或低配资源训练 Prompt、流程和验收标准，不完全依赖最强模型。
  5. 为老业务列 P0 清单：只处理挂了、大故障、客户严重受影响的事项。
  6. 新业务先由核心人拿到突破证据，再决定是否迁移旧团队成员。
  7. 面试业务岗时不要只问执行经验，要追问他为什么这样判断、如何处理不确定性。
  8. 开发岗位优先找 2-3 年经验但已有底层判断雏形的人，用真实任务观察 6 个月。
  9. 为关键词推广建立观察表：搜索量、展示、点击、展示占比、转化和内容缺口。
  10. 对每个 Agent 项目写清 AI 边界：模型能做什么，业务仍必须由谁判断。
