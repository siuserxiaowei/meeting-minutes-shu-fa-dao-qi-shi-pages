业务发展及招人策略讨论

AI组织增长产品商业化

这场讨论把项目托管、飞书协作、MCP/Codex、第二曲线、招聘和推广放在一起，重点不是单一工具，而是组织如何把上下文、业务优先级和人能力练出来。核心判断是：老业务只处理 P0，新业务要独立突破；招聘看“道”的潜力，工具和低配资源用于倒逼流程成熟。

道法术器势拆解

道 底层原则、价值判断、长期认知。

核心判断
组织真正要建设的是判断力和上下文，而不是堆工具。招聘看“道”，协作看同步，业务看优先级；AI 只是在这个系统里放大人的判断和流程。
证据依据
纪要明确提出招聘关注“道”的层面、每日固定同步上下文，以及“只有确定 AI 的边界，才能确保下一个产品不被碾压”。
可复用动作
每次招人或上新工具前，先问：它提升的是判断力、协作密度，还是只是增加了一个操作界面。

法 可复用方法、流程、判断框架。

核心判断
可复用管理方法是“话题群承载上下文 -> 低配资源倒逼流程 -> 老业务 P0 化 -> 新业务独立突破 -> 合适人员再迁移”。
证据依据
会议决策包括迁移飞书、配置话题群、低配资源训练人员、业务切割、老带新和试用期培养。
可复用动作
为每条业务设定三层规则：信息同步节奏、P0 判定标准、人员迁移条件。

术 可执行动作、技巧、话术、步骤。

核心判断
当下动作很具体：迁移飞书、建话题群、接入 MCP/Codex、限定老业务处理时间、制定招聘面试题和关键词观察表。
证据依据
纪要中的关键决策是迁移飞书并配置话题群；同时提到用 Codex 辅助运营查询数据、关注 humanizeAI/bypass 等关键词。
可复用动作
本周落地：建 3 个核心话题群，每个话题固定 owner、目标、资料入口、每日同步时间和 P0 清单。

器 工具、资源、模板、渠道、技术栈。

核心判断
器包括 GitHub、云部署、TPR、飞书话题群、MCP、Codex、低配模型资源、B 站和关键词数据表。工具要被设计成训练组织能力的环境。
证据依据
讨论中工具不是孤立出现，而是用于项目托管、内部数据查询、办公入口、推广观察和团队训练。
可复用动作
给每个工具标注“训练什么能力”：查询、同步、决策、推广、交付或复盘；不能标注的工具先不引入。

势 趋势、外部环境、机会窗口、结构性变化。

核心判断
老业务现金流不稳定、AI 能力持续外溢、热门关键词快速变化，共同推动团队寻找第二曲线。未来竞争不只看技术，而看组织能否快速同步、判断和转向。
证据依据
会议把业务现金流不稳定、新业务突破、AI 边界、关键词流行度和招聘标准放在同一张图里讨论。
可复用动作
每月复盘一次“势”：老业务风险、新业务窗口、AI 边界变化、流量关键词变化和团队能力缺口。

章节索引

  1. 00:00 项目托管、部署与内部工具讨论 GitHub、云部署、MCP 和 Codex，用工具让运营人员自行查询数据。
  2. 00:45 迁移飞书与话题群协作把飞书作为第一入口，一个话题对应一个 session，推动信息同步和流程沉淀。
  3. 04:01 业务转型与第二曲线老业务现金流不稳定，需要切割精力，只处理 P0 问题，发展新业务。
  4. 07:54 招聘用人标准招聘时优先看候选人对“道”的理解和潜力，而不是只看具体技术动作。
  5. 20:42 关键词流行度与推广关系通过关键词流行度、搜索量、曝光和点击之间的关系估算推广机会。

行动项

  1. 把工作入口迁移到飞书，按业务话题建立群组和固定讨论 session。
  2. 老业务每天限定固定时间，只处理 P0 问题，把精力释放给新业务验证。
  3. 用 MCP 和 Codex 做内部数据查询入口，让运营人员自己完成基础分析。
  4. 招聘时增加“道”的面试题：让候选人解释业务判断、资源取舍和长期边界。
  5. 用关键词流行度和搜索量建立推广观察表，持续跟踪可借势的流量窗口。
